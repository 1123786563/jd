## ✅ 已实施的优化

### 1. 使用共享日志函数 ✅

**改动**:
- 删除了本地 `log` 函数的重复实现
- 导入并使用 `src/lib/logging.ts` 中的共享 `log` 函数

**代码变化**:
```typescript
// 之前
function log(level: string, message: string): void {
    const timestamp = new Date().toISOString();
    const logMessage = `[${timestamp}] [${level}] ${message}\n`;
    console.log(logMessage.trim());
    fs.appendFileSync(LOG_FILE, logMessage);
}

// 之后
import { log } from '../lib/logging';
// 直接使用
log('INFO', 'Message');
```

**收益**:
- 减少代码重复
- 统一日志格式
- 便于全局日志策略调整

### 2. 使用共享配置常量 ✅

**改动**:
- 删除了 `TINYCLAW_HOME`、`SETTINGS_FILE`、`FILES_DIR` 的重复定义
- 导入并使用 `src/lib/config.ts` 中的共享常量

**代码变化**:
```typescript
// 之前
const TINYCLAW_HOME = process.env.TINYCLAW_HOME
    || (fs.existsSync(path.join(_localTinyclaw, 'settings.json'))
        ? _localTinyclaw
        : path.join(require('os').homedir(), '.tinyclaw'));
const SETTINGS_FILE = path.join(TINYCLAW_HOME, 'settings.json');
const FILES_DIR = path.join(TINYCLAW_HOME, 'files');

// 之后
import { SETTINGS_FILE, FILES_DIR, TINYCLAW_HOME } from '../lib/config';
```

**收益**:
- 确保所有客户端使用相同的配置路径
- 减少错误风险
- 便于配置集中管理

## 🔍 其他发现（未实施）

在审查过程中还发现了以下可以进一步优化的地方：

### 1. 命令处理逻辑重复 ⚠️

`/agent`、`/team`、`/reset` 命令的处理在所有客户端中重复，建议提取到共享库。

### 2. 资源清理优化 ⚠️

`pendingMessages` 清理逻辑可以在 `handleEvent` 外部统一处理，改为定期清理。

### 3. 访问令牌管理 ⚠️

当前使用全局变量，建议封装为类以更好地管理并发和状态。

### 4. 错误处理统一 ⚠️

建议创建共享的错误处理器，统一错误日志级别和格式。

### 5. 常量提取 ⚠️

建议提取硬编码的字符串（如 `"p2p"`、`"text"`）为常量或枚举。
